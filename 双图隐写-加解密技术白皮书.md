# 双图隐写加解密技术白皮书

版本：1.1  
日期：2026-06-21  
适用程序：`double-image-secret.html`

## 摘要

“双图隐写”是一个离线单页 Web 工具，用两张图片共同承载一段短文字。它的核心安全机制不是传统的密码口令或 AES 加密，而是“随机 XOR 双份额拆分 + 图像隐写 + 纠错校验”的组合方案。

写入时，程序先把短文字封装成固定长度 payload，再生成一份随机字节序列作为份额 A，并用 `payload XOR shareA` 得到份额 B。随后两份数据分别被写入两张图片的 DCT 中频系数中。读取时，程序必须同时从两张图片提取份额，并执行 XOR 合并，才能恢复 payload 和原始文字。

该方案的目标是日常场景下“不显眼、离线、两图缺一不可”，而不是面向专业取证对抗、强密码学认证或长期高价值秘密保存。

## 1. 系统定位

### 1.1 设计目标

本程序面向短暗号、日期、名称缩写等小容量文本隐藏需求，主要目标如下：

- 本地处理：图片和文字均在浏览器内处理，不上传服务器。
- 双图门槛：单张图片不能直接还原明文，必须两张图共同参与。
- 视觉不显眼：写入后的图片肉眼尽量无明显变化。
- 基础抗扰动：对 PNG 保存、高质量 JPG 压缩和普通头像裁剪有一定容忍度。
- 失败可检测：通过 Header 和 CRC32 判断图片错配、损坏或压缩过强。

### 1.2 非目标

本程序不承诺以下能力：

- 不提供用户口令、密钥派生、AES、RSA、ECC 等传统密码系统。
- 不提供篡改认证或抗主动攻击能力。
- 不保证能抵抗强压缩、重裁剪、滤镜、截图转发或只保留局部区域的平台处理。
- 不保证能抵抗专业隐写分析或取证检测。
- 不适合保存高价值、长期敏感或法律合规要求严格的秘密。

## 2. 加解密总体架构

系统由四个主要层次构成：

1. 文本封装层：把 UTF-8 短文字封装为固定 50 字节 payload。
2. 双份额拆分层：使用随机字节和 XOR 把 payload 拆成两份。
3. 纠错编码层：使用 Hamming(7,4) 和重复映射提高读取容错。
4. 图像隐写层：在 768x768 归一化图像的 8x8 DCT 中频块中写入 bit。

写入流程：

```text
短文字
  -> UTF-8 bytes
  -> payload(magic/version/length/body/crc32)
  -> shareA = random(50 bytes)
  -> shareB = payload XOR shareA
  -> shareA/shareB 分别转 bit
  -> Hamming 纠错编码
  -> 重复映射
  -> DCT 中频系数写入图片 A/B
  -> 导出 PNG/JPG
```

读取流程：

```text
图片 A/B
  -> 生成完整图、中心裁剪和轻微偏移裁剪候选
  -> 按 crop-v2 或 legacy-v1 块序扫描 DCT 中频系数差
  -> 重复块投票
  -> Hamming 纠错解码
  -> 还原 shareA/shareB
  -> payload = shareA XOR shareB
  -> Header/CRC32/UTF-8 检查
  -> 输出短文字
```

## 3. Payload 数据格式

程序把明文封装为固定 50 字节 payload。固定长度有两个作用：一是简化嵌入容量规划，二是避免从图像载荷长度上直接暴露明文长度以外的结构变化。

| 字节范围 | 长度 | 字段 | 说明 |
|---|---:|---|---|
| 0-3 | 4 | Magic | 固定为 `0x51 0x50 0x31 0x21`，即 `QP1!` |
| 4 | 1 | Version | 当前写入为 `2`，读取兼容 `1` |
| 5 | 1 | Length | 正文 UTF-8 字节长度，最大 40 |
| 6-45 | 40 | Body | UTF-8 正文，不足部分保留为 0 |
| 46-49 | 4 | CRC32 | 对字节 0-45 计算 CRC32，按大端序写入 |

核心参数：

| 参数 | 值 | 含义 |
|---|---:|---|
| `MESSAGE_BYTES` | 40 | 最大正文长度，按 UTF-8 字节计算 |
| `PAYLOAD_BYTES` | 50 | payload 固定总长度 |
| `HEADER_BYTES` | 6 | magic、version、length 的总长度 |
| `CRC_OFFSET` | 46 | CRC32 起始位置 |
| `CURRENT_VERSION` | 2 | 当前写入载荷版本 |
| `LEGACY_VERSION` | 1 | 兼容读取的旧载荷版本 |

CRC32 只用于错误检测，不是密码学认证码。它可以发现常见的错配、破坏和读取错误，但不能防止有意伪造。

## 4. 双份额 XOR 机制

### 4.1 写入侧拆分

程序生成 50 字节随机 `shareA`，再计算：

```text
shareB = payload XOR shareA
```

因此：

```text
payload = shareA XOR shareB
```

两张图分别承载 `shareA` 和 `shareB`。任何一张图单独被读取时，只能得到一段看似随机的 50 字节数据，不能直接恢复 payload。

### 4.2 随机数来源

首选随机数来源是浏览器 Web Crypto API：

```text
window.crypto.getRandomValues()
```

如果浏览器不支持该 API，代码会退回到 `Math.random()`。从安全性角度看，`Math.random()` 不适合作为密码学随机数来源。因此，严肃使用场景应要求运行环境支持 Web Crypto，并可考虑移除弱随机 fallback。

### 4.3 安全性质

在 `shareA` 真随机、长度等于 payload、且只使用一次的前提下，单独持有 `shareA` 或 `shareB` 不会泄露 payload 信息。这类似一次性密码本或 2-of-2 秘密分享。

但该性质有明确边界：

- 同时拿到两张图即可恢复明文，没有额外口令或密钥。
- 如果随机数质量不足，份额可能存在可预测风险。
- 如果同一份随机 share 被复用到多个 payload，会破坏安全性。当前实现每次写入重新生成随机 share，未主动复用。
- Header、长度和 CRC32 在两份合并后可验证，但没有防伪造认证。

## 5. 纠错编码与重复映射

图像压缩、量化、色彩处理会导致嵌入 bit 读取错误。程序使用两层冗余提高稳定性。

### 5.1 Hamming(7,4)

每 4 个原始 bit 被编码为 7 个 bit：

```text
[p1, p2, d1, p4, d2, d3, d4]
```

其中：

```text
p1 = d1 XOR d2 XOR d4
p2 = d1 XOR d3 XOR d4
p4 = d2 XOR d3 XOR d4
```

读取时根据 syndrome 定位并修正单 bit 错误，再恢复 `[d1, d2, d3, d4]`。

### 5.2 分层重复写入

`crop-v2` 每个 Hamming 编码 bit 会写入 9 个不同图像块，其中多数点优先落在头像中心安全区域，其余点分散到中区和全图。读取时对可用重复块的 DCT 系数差进行累加投票：

```text
score >= 0 -> bit 1
score < 0  -> bit 0
```

该机制能缓解局部噪声、压缩损伤和个别块读取错误。

### 5.3 容量计算

单份额大小为 50 字节，即 400 bit。Hamming(7,4) 后：

```text
400 / 4 * 7 = 700 bit
```

当前 `crop-v2` 每个编码 bit 重复 9 次，因此每张图需要：

```text
700 * 9 = 6300 个 8x8 图像块
```

归一化图像为 768x768，按 8x8 分块：

```text
96 * 96 = 9216 个块
```

可用块数量大于所需块数量，容量上满足写入需求。旧版 `legacy-v1` 使用 7 次重复，即每张图 4900 个块。

## 6. 图像隐写方法

### 6.1 图像归一化

写入时，所有输入图片都会被中心裁切为正方形，再缩放到 768x768。这样可以固定分块结构、减少不同尺寸图片对读取结果的影响。

读取时，新版会生成完整图、常见中心裁剪、轻微偏移裁剪和轻微比例变化候选，再通过 CRC32 判断哪组候选有效。该设计提升了普通头像平台裁剪后的读取成功率，但如果外部平台重裁剪、滤镜、强压缩或只保留局部区域，仍可能失败。

### 6.2 DCT 中频系数写入

程序把图像划分为 8x8 块，对每个块的亮度值执行 DCT。写入时使用两个中频系数：

```text
COEFF_A = [3, 4]
COEFF_B = [4, 3]
```

判断与写入依据是两个系数的差：

```text
diff = DCT[3,4] - DCT[4,3]
```

写入 bit 的规则：

- 写入 `1`：确保 `diff >= strength`
- 写入 `0`：确保 `diff <= -strength`

如果当前差值未达到目标强度，程序会对两个系数做对称调整，再执行逆 DCT，把亮度差回写到 RGB 通道。

### 6.3 写入强度

界面提供三档稳定度：

| 档位 | strength | 取舍 |
|---|---:|---|
| 轻微 | 18 | 视觉改动更小，但抗压缩能力较弱 |
| 标准 | 26 | 默认平衡项 |
| 增强 | 34 | 读取稳定性更强，但图像改动更明显 |

`crop-v2` 写入会在所选档位上做温和增强，以抵消裁剪和重采样带来的中频损伤。强度越高，中频系数差越容易在压缩后保留下来，但也越可能引入轻微视觉变化或增大被分析发现的概率。

### 6.4 块位置伪随机化

旧版使用固定种子 `0x51f15e7d` 和 xorshift32 对全部 8x8 块索引洗牌，然后取前 4900 个块作为嵌入位置。新版 `crop-v2` 改为分层块池：中心安全区、中间区域和全图区域分别洗牌，每个编码 bit 按固定比例从不同块池取 9 个位置。

该做法能避免信息集中写入图像某一区域，也让损伤分布更平均。但种子写死在代码中，不构成秘密密钥；任何拿到程序的人都能复现同一块序。

## 7. 读取与校验

读取时，程序对每张图生成多组裁剪候选。候选会先走完整图和常见中心裁剪，失败后再进入轻微偏移和更深候选搜索。每个重复块读出 DCT 中频系数差，累加投票得到编码 bit，再经 Hamming 解码还原份额。

合并后的 payload 依次经过以下检查：

1. Magic 检查：确认前 4 字节为 `QP1!`。
2. Version 检查：确认版本号为当前 `2` 或兼容旧版 `1`。
3. Length 检查：确认正文长度不超过 40 字节。
4. CRC32 检查：确认 payload 字节 0-45 与存储校验值一致。
5. UTF-8 检查：使用 fatal 模式解码，发现非法字节即失败。

这些检查能有效区分“成功读取”和“图片错配或破坏”。例如两张不匹配图片通常会在 Header 或 CRC32 阶段失败。

## 8. 安全分析

### 8.1 保密性

保密性主要来自 XOR 双份额拆分。只要随机份额不可预测，单张图片中的载荷在理论上不透露明文。

实际使用中，明文安全取决于：

- 两张图是否被分开保存和传输。
- 运行环境是否支持强随机数。
- 攻击者是否同时获得两张图和读取工具。
- 图像是否被平台处理到无法读取，导致可用性下降。

### 8.2 完整性

CRC32 能检测随机错误、图片错配和常见损坏，但它不是抗攻击的完整性保护。攻击者如果理解 payload 格式并能重新写入两张图，理论上可以构造新的有效载荷。

如需抗篡改，应引入 HMAC 或 AEAD，例如 `AES-GCM`，并配合用户口令或外部密钥派生。

### 8.3 隐蔽性

DCT 中频写入比直接修改像素 LSB 更适合对抗普通 JPEG 压缩，也较不容易被肉眼发现。但由于写入位置、系数选择和算法均公开，面对专业隐写分析时不能视为不可检测。

### 8.4 可用性风险

读取成功率容易受以下因素影响：

- 平台压缩质量过低。
- 普通头像裁剪一般会自动候选搜索；重裁剪、比例变化过大或只保留局部区域仍可能失败。
- 添加滤镜、美颜、锐化、降噪等处理。
- 截图、转码或社交平台重新编码。
- 用户只保存了其中一张图，或 A/B 图来自不同写入批次。

## 9. 威胁模型

| 场景 | 预期结果 |
|---|---|
| 攻击者只有一张生成图 | 理论上无法恢复 payload |
| 攻击者同时获得两张生成图和本程序 | 可以读取明文 |
| 攻击者获得两张图但不知道本程序 | 仍可能通过逆向或复现算法读取 |
| 图片被轻度 JPEG 压缩 | 通常可读取，取决于质量和强度档位 |
| 图片被普通头像裁剪 | 通常会自动尝试读取，实测 `90%`、`84%`、`78%` 中心裁剪和 `80%-84%` 轻微偏移裁剪可读 |
| 图片被重裁剪、滤镜或强压缩 | 可能读取失败 |
| 攻击者主动伪造两张图片 | CRC32 不足以提供密码学防伪 |

## 10. 与传统加密的区别

本程序的“解密”并不是输入密钥后执行密码算法解密，而是“两份额合并”：

```text
payload = shareA XOR shareB
```

因此它更接近 2-of-2 秘密分享。安全边界是“两张图必须分离”，不是“密码必须保密”。这一区别对使用者很重要：两张图一旦同时泄露，隐藏文字就不再有密码学保护。

## 11. 改进建议

如果需要提升安全级别，建议按优先级考虑以下改造：

1. 增加口令加密：在 payload 拆分前使用 `AES-GCM` 加密正文。
2. 增加密钥派生：使用 `PBKDF2`、`scrypt` 或 `Argon2id` 从用户口令派生密钥，并保存 salt。
3. 增加认证保护：使用 AEAD tag 或 HMAC 替代 CRC32 的安全职责。
4. 移除弱随机 fallback：不支持 Web Crypto 时直接拒绝写入。
5. 引入密钥化块序：用口令派生的 seed 控制块位置，避免所有用户共享固定嵌入路径。
6. 增加格式冗余：为 header 使用多份编码，提升强压缩后定位和版本识别能力。
7. 增加读取诊断：输出纠错统计、投票置信度和失败阶段，便于判断图片损伤程度。

## 12. 结论

“双图隐写”的现有方案在日常使用中具有清晰、轻量的安全边界：单张图不够，两张图合并才可读；所有处理本地完成；图片外观尽量保持自然。它适合隐藏低到中敏感度的短文本，不适合承载高价值秘密或对抗专业攻击。

从技术上看，随机 XOR 双份额提供了单份额保密性，Hamming 纠错和重复映射提供了读取稳健性，DCT 中频写入提供了相对自然的图像承载方式。后续如果引入口令加密、AEAD 认证和密钥化嵌入路径，可以把系统从“日常隐蔽工具”提升为更接近完整密码系统的安全设计。
